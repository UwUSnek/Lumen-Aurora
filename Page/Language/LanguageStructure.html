After the preprocessing phase, only a small number of characters are left.
These characters compose the syntactic elements that are used in the compilation phase.
Any unknown character generates a syntactic error. //TODO LINK

String literals can contain any unicode character. They are converted to allowed characters during the compilation phase.

Syntactic elements, also known as Tokens, are the fundamental elements of the source code of any program.
They are sequences of one or more allowed characters that follow specific patterns.
One or more syntactic elements create a semantic element.

A syntactic element ends when a character that isn't allowed in its pattern is encountered. Syntactic elements can also be separated by any number of , and characters or comments.
Those who that don't follow any of the specified patterns generate a syntactic error. //TODO LINK //TODO sizeof

Syntactic elements are case sensitive, with the only exception of hexadecimal literals.
, and characters are not syntactic elements.

Keywords are built-in syntactic elements that perform hard coded actions or have special meaning.
The language defines a total of 24 alphanumeric keywords and 5 symbolic keywords.
Their usage is detailedly explained in their respective sections.

  • Semantic keywords
    • namespace Used to declare a namespace.
    • enum Used to declare an enum.
    • struct Used to declare a struct.
    • auto Can be used in place of the type of a variable or routine parameter to declare it using the type of its initializer value.
    • typeof Retrieves the type of an expression.
    • alias Creates an alternative path for a symbol.
    • import Imports symbols from a module.
    • export Exports symbols from the module.

  • Flow control keywords
    • if Changes execution path based on one condition.
    • match Changes execution path based on the value of an expression.
    • for Executes a statement an arbitrary number of times.
    • while Executes a statement until its condition evaluates to false.
    • do Used with while to check the condition after having executed the statement instead of the other way around.
    • continue Skips an arbitrary number of iterations of the innermost enclosing loop.
    • break Skips all the remaining iterations of an arbitrary number of enclosing loops, starting from the innermost.
    • once Executes a statement only during the first iteration of the innermost enclosing loop.
    • then Executes a statement if the immediatly preceeding loop executed at least one iteration and didn't stop through a break statement.
    • else Executes a statement if the condition of the immediatly preceeding loop or if statement evaluates to false.
    • return Returns the execution to the caller function.
    • defer Delays a routine call until the first return statement is encountered.

  • Containers
    • () Groups an expression, or encloses a list of routine parameters. Also used in operator precedence and template constraints.
    • {} Declares a new scope. Used to declare most semantic elements.

  • Separators
    • , Separates 2 or more expressions.
    • ; Defines the end of expression statements or certain declarations and directives.
    • . Concatenates 2 ore more local names to form a symbol path.

Literals are a special type of syntactic element, as they can contain both alphanumeric characters and ..
Their usage and syntax are explained in Literals.

Identifiers are used to name and reference symbols.
An identifier can be alphanumeric ([a-zA-Z_][a-zA-Z_0-9]*) or symbolic ([!$%&()*+\-/:<=>?@^`|~;{}\[\]\.]+).

All identifiers must be unique to their scope.
Declaring more than one symbol with the same identifier in the same scope is a semantic error.

Semantic elements are sequences of one or more syntactic elements that follow specific patterns defined by the syntax of the language.
They define the contents and interface of the module.

Certain semantic elements can be referenced to retrieve their value or obtain additional informations. These elements are called symbols.
Symbols are referenced through their name, or using the typeof specifier.
Unnamed types are also symbols. //FIXME

  • Semantic element -
    • Declaration Defines the interface and behaviour of a user-defined semantic element.
      • Namespace declaration Defines a named scope.
      • Enum declaration Groups compile time values of the same type.
      • Struct declaration Groups variables of different types.
      • Function declaration A routine that requires parentheses to be called.
      • Operator declaration A routine that doesn't require parentheses to be called and can use multiple symbolic or alphanumeric identifiers.
      • Template declaration Generates declarations of semantic elements.
        • Enum template Generates enums based on the explicit template arguments.
        • Struct template Generates structs based on the explicit template arguments.
        • Function template Generates functions based on the explicit template arguments or the types of the argumets used in the routine call.
        • Operator template Generates operators based on the explicit template arguments or the types of the argumets used in the routine call.
      • Symbol alias declaration Defines an alternative path for a symbol.
      • Variable declaration Identifies one or more named memory locations that can store values.
    • Statement Instructions executed in runtime when a routine is called
      • if statement Executes a statement if its condition evaluates to true.
      • match statement Executes statements based on the value of an expression.
      • while loop Executes a statement until its condition evaluates to false.
      • for loop Executes a statement a fixed amount of times.
      • once statement Executes a statement during the first iteration of the innermost enclosing loop.
      • then statement Executes a statement if the immediatly preceeding loop has not stopped through a break statement and has completed at least one iteration.
      • break statement Skips an arbitrary number of iterations of the innermost enclosing loop.
      • continue statement Skips all the remaining iterations of an arbitrary number of enclosing loops, starting from the innermost.
      • else statement Executes a statement if the immediatly preceeding loop or if statement has not completed any iteration.
      • return statement Returns the flow control to the caller routine.
      • defer statement Executes a statement immediatly before the first return statement is executed.
      • Expression statement Evaluates an expression.
      • Unnamed scope Groups multiple statements in one.
      • Empty statement Does nothing.
      • Inline GLSL Allows GLSL 4.60 code to be used from within Lux.
      • Inline C++ Allows C++20 code to be used from within Umbra.
    • expression Generates a value when evaluated
      • Literal Generates a inline value of a primitive type.
      • Constructor Generates an inline value of any type.
      • Variable Accesses the value stored in a previously declared variable.
      • Temporary variable Accesses the value stored in a temporary variable.
      • Function call Generates a value by calling a function or function template.
      • Operator call Generates a value by calling an operator or operator template.
      • try expression Compiles a portion of code. Evaluates to true if the code doesn't generate syntactic or semantic errors.
    • import directive Imports symbols from a module.
    • export directive Defines the symbols the module provides when imported.

  • Semantic sub-element Parts of complex semantic elements that are split in order to define better syntax rules
    • Template constraint Defines the constraint that controls when the subsequent element body is available
    • Routine parameters Defines the input values of a routine
    • Enum element declaration Defines one or more enum elements and their values.
    • Struct member declaration Defines one ore more struct members.
    • Symbol path References a symbol

  • Semantic pseudo-element -
    • While signature Contains the condition and structure of the while statements
    • Template signature Defines the name and parameters of a template
    • Function signature Defines the name, return type and parameters of a function
    • Operator signature Defines the names, return type, parameters and priority of an operator
    • Routine body Contains the statements and the children symbols of a routine
    • Enum signature Defines the name of an enum and the primitive type it derives from.
    • Enum body Contains the enum elements and other children symbols.
    • Struct signature Defines the name of a struct.
    • Struct body Contains the struct members and other children symbols.

Symbols are referenced through their name.
A path is the list of all the local names used to refer to the enclosing scopes and the symbol separated by . tokens, starting from the outermost.
The local name depends on the type of semantic element the symbol is declared as. They are described in the corresponding sections of each semantic element and summarized in the Local names section.

The name can start from any of the enclosing scopes.
If the enclosing scopes are omitted, the symbol identifier is searched in the current scope.
Names that start from the module scope are called complete names. Names that omit the enclosing scopes are called local names. Other names are called relative names.

local name .
Example
// Complete name: module // Complete name: module.s // Local name: s namespace s { // Absolute name: module.s.fun // Relative names: s.fun // Local name: fun void fun(uint n) { // ^ // Absolute name: module.s.fun.n // Relative names: s.fun.n, fun.n // Local name: n } } namespace { // No complete name // No relative names // Local name: n uint n; }

The same identifier can be used to declare multiple symbols in different scopes.
When the scopes are one inside another and the identifier is used in relative or local names from one of the inner scopes, the innermost symbol takes precedence over the others.
This is known as name shadowing.

Absolute names cannot be shadowed

Example
namespace s { // n references module.s.n void n(uint n) { // n references module.s.n.n namespace { float n; // n references module.s.n.*unnamed*.n } } }

Any unknown value of any type is an undefined value.
The language can generate undefined values for a variety of reasons.
Such values are usually the result of errors that occurred during the execution of a function and don't contain any useful information.
Uninitialized variables also have undefined value.

Generating undefined values doesn't stop the execution and is not signaled in any way, nor is it detectable.
The module is responsible for preventing, discarding or documenting undefined values.

Certain actions cause undefined behaviour.
Undefined behaviour means that the code compiles with no erros, but the runtime behaviour is unknown and anything could happen.
The module could work just fine, or it could break and never run again. No one knows.

As with undefined values, the module is responsible for preventing or documenting undefined behaviours.

Compile time values are values that can be calculated during the compilation phase.
Whether a value can be calculated or not depends on precise rules:

  • It cannot depend on parameters of the main function
  • It cannot depend on

A value a is said to depend on a value b if b is used in one of the preceeding function calls where a is passed as argument of a reference parameter.

Known special cases that generate constant values, such as n * 0 or n - n, are used by the compiler to optimize the code, but for consistency reasons, they have no relevance in determining if a value is compile time or not.
